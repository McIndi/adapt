## Executive Summary

This review examines three critical areas of the Adapt codebase following the recent Phase 2 authentication and authorization implementation. The codebase shows good architectural design and solid fundamentals, but several issues could lead to production problems, particularly around **database session management**, **session expiration enforcement**, and **race conditions in locking**.

**Overall Assessment:** The code is well-structured but requires attention to resource management and concurrency edge cases before production deployment.

## Review Area 1: Database Session Management & Resource Leaks

### Severity: **HIGH** ⚠️

### Issue Summary

The codebase exhibits **inconsistent database session lifecycle management** across multiple modules, creating potential for connection leaks, resource exhaustion, and database locking issues under concurrent load.

### Specific Problems

#### 1.1 Session Leak in `auth.py::get_current_user` (Lines 54-65)

**Location:** [auth.py:54-65](file:///c:/Users/cliff/projects/adapt/adapt/auth.py#L54-L65)

```python
def get_current_user(request: Request) -> User | None:
    token = request.cookies.get(SESSION_COOKIE)
    if not token:
        return None
    
    with get_db_session(request) as db:  # ✅ Uses context manager
        session = get_session(db, token)
        if not session:
            return None
        return db.get(User, session.user_id)  # ❌ Returns detached object!
```

**Problem:**  
The function returns a [User](file:///c:/Users/cliff/projects/adapt/adapt/storage.py#30-37) object obtained from the database session, but **exits the context manager immediately**, causing the object to become **detached**. Any lazy-loaded attributes or relationships will fail when accessed later.

**Impact:**
- Potential `DetachedInstanceError` exceptions when accessing user attributes
- Breaks SQLModel relationship loading (e.g., if you added `relationships` to the User model)
- Silent failures in production

**Why This Matters:**  
Multiple request handlers depend on [get_current_user](file:///c:/Users/cliff/projects/adapt/adapt/auth.py#54-66) via FastAPI's dependency injection system. If the returned User object is accessed for relationships (groups, permissions indirectly), it will crash.

#### 1.2 Missing Session Cleanup in [permission_dependency](file:///c:/Users/cliff/projects/adapt/adapt/auth.py#97-121) (Lines 97-120)

**Location:** [auth.py:97-120](file:///c:/Users/cliff/projects/adapt/adapt/auth.py#L97-L120)

```python
def permission_dependency(action_param: str, resource: str):
    async def check(request: Request, user: User = Depends(require_auth)):
        action = action_param
        if action == "auto":
            # ... action determination logic

        db = Session(request.app.state.db_engine)  # ❌ Manual session creation
        try:
            if not check_permission(user, db, action, resource):
                 raise HTTPException(...)
            return user
        finally:
            db.close()  # ⚠️ Close but no rollback on exception
    return check
```

**Problems:**
1. **No transaction rollback**: If [check_permission()](file:///c:/Users/cliff/projects/adapt/adapt/auth.py#77-96) raises an exception, the session closes without rolling back, potentially leaving a dirty transaction state
2. **Inconsistent pattern**: Uses manual [Session()](file:///c:/Users/cliff/projects/adapt/adapt/storage.py#22-29) creation instead of context manager (contrast with [get_current_user](file:///c:/Users/cliff/projects/adapt/adapt/auth.py#54-66))
3. **No connection pool management**: Relies on manual close, which is error-prone

**Impact:**
- Connection leaks if exception occurs before `finally` block
- Potential transaction deadlocks under high concurrency
- Database connection pool exhaustion

#### 1.3 Detached Object Pattern in [admin.py](file:///c:/Users/cliff/projects/adapt/adapt/admin.py)

**Location:** [admin.py:47-50](file:///c:/Users/cliff/projects/adapt/adapt/admin.py#L47-L50), [admin.py:111-127](file:///c:/Users/cliff/projects/adapt/adapt/admin.py#L111-L127)

```python
@router.get("/users", response_model=List[User])
def list_users(request: Request, user: User = Depends(require_superuser)):
    with get_db(request) as db:
        return db.exec(select(User)).all()  # ❌ Returns detached objects
```

**Problem:**  
All admin endpoints return database objects immediately after the context manager exits, making them detached. While this works for simple serialization, it breaks if:
- You add relationships to models (User → Sessions, Groups → Permissions)
- You implement lazy loading
- Pydantic tries to access SQLModel relationships during serialization

**Current State:** Works by accident because models are simple  
**Future Risk:** Will break when relationships are added

### Recommendations

#### Fix 1: Use Proper Session Scoping with FastAPI Dependencies

Replace manual session management with FastAPI's dependency injection:

```python
# In storage.py or a new db.py module
from sqlmodel import Session
from typing import Generator

def get_db_session(request: Request) -> Generator[Session, None, None]:
    """FastAPI dependency for database sessions."""
    db = Session(request.app.state.db_engine)
    try:
        yield db
        db.commit()  # Commit successful operations
    except Exception:
        db.rollback()  # Rollback on error
        raise
    finally:
        db.close()  # Always cleanup

# Usage in routes
@router.get("/users")
def list_users(db: Session = Depends(get_db_session)):
    return db.exec(select(User)).all()  # Objects stay attached during request
```

#### Fix 2: Explicitly Load Data Before Session Closes

For [get_current_user](file:///c:/Users/cliff/projects/adapt/adapt/auth.py#54-66), eagerly load the user within the session:

```python
def get_current_user(request: Request) -> User | None:
    token = request.cookies.get(SESSION_COOKIE)
    if not token:
        return None
    
    with get_db_session(request) as db:
        session = get_session(db, token)
        if not session:
            return None
        user = db.get(User, session.user_id)
        if user:
            # Force load any relationships before detaching
            db.refresh(user)
        return user
```

#### Fix 3: Use Context Managers Consistently

**Replace all instances of:**
```python
db = Session(engine)
try:
    # work
finally:
    db.close()
```

**With:**
```python
with Session(engine) as db:
    try:
        # work
        db.commit()
    except:
        db.rollback()
        raise
```

### Testing Recommendations

1. Add load test with 100 concurrent requests to verify no connection leaks
2. Monitor connection pool metrics (active connections, pool exhaustion)
3. Add integration test that accesses User relationships after authentication
4. Test exception paths to ensure sessions are properly cleaned up

## Review Area 2: Session Expiration & Security

### Severity: **MEDIUM-HIGH** ⚠️

### Issue Summary

Session management has multiple security and correctness issues:
1. **Session expiration is never enforced** after creation
2. **No session cleanup background task**
3. **Timing attack vulnerability** in session validation
4. **Missing Secure/SameSite cookie flags**

### Specific Problems

#### 2.1 Session Expiration Not Enforced (Lines 46-48)

**Location:** [auth.py:46-48](file:///c:/Users/cliff/projects/adapt/adapt/auth.py#L46-L48)

**Code:**
```python
def get_session(db: Session, token: str) -> DBSession | None:
    stmt = select(DBSession).where(DBSession.token == token)
    return db.exec(stmt).first()  # ❌ NEVER checks expires_at!
```

**Problem:**  
Sessions are created with a 7-day TTL (`SESSION_TTL = timedelta(days=7)`, line 29), and `expires_at` is set during creation (line 38), but **get_session() never validates the expiration**.

**Impact:**
- **Sessions never expire** — users stay logged in forever
- Security risk: stolen session tokens remain valid indefinitely
- Contradicts the spec which claims "7-day session expiration with automatic cleanup"

**Proof:**
```python
# auth.py:31-44
def create_session(db: Session, user_id: int) -> str:
    token = secrets.token_urlsafe(32)
    now = datetime.now(tz=timezone.utc)
    session_obj = DBSession(
        user_id=user_id,
        token=token,
        created_at=now,
        expires_at=now + SESSION_TTL,  # ✅ Sets expiration
        last_active=now,
    )
    # ... commit

# But get_session() ignores it:
def get_session(db: Session, token: str) -> DBSession | None:
    stmt = select(DBSession).where(DBSession.token == token)
    return db.exec(stmt).first()  # ❌ Should filter by expires_at
```

#### 2.2 No Session Cleanup Task

**Problem:**  
Expired sessions accumulate in the database indefinitely. There's no background task or cron job to delete them.

**Impact:**
- Database bloat (sessions table grows without bounds)
- Performance degradation on session lookups
- Potential PII retention issues (stored session data longer than necessary)

**Contrast with Locks:**  
The [LockManager](file:///c:/Users/cliff/projects/adapt/adapt/locks.py#11-90) has [release_stale_locks()](file:///c:/Users/cliff/projects/adapt/adapt/locks.py#60-68) (locks.py:60), but there's no equivalent for sessions.

#### 2.3 Timing Attack in Session Lookup

**Location:** [auth.py:46-48](file:///c:/Users/cliff/projects/adapt/adapt/auth.py#L46-L48)

**Problem:**  
The session lookup uses `.first()`, which uses database comparison. An attacker can measure response times to determine if a token exists in the database vs. not existing.

**Fix:** Add constant-time comparison after retrieval.

#### 2.4 Insecure Cookie Configuration

**Location:** [auth.py:142](file:///c:/Users/cliff/projects/adapt/adapt/auth.py#L142)

```python
response.set_cookie(
    key=SESSION_COOKIE, 
    value=token, 
    httponly=True,  # ✅ Good
    max_age=int(SESSION_TTL.total_seconds())
    # ❌ MISSING: secure=True, samesite='lax'
)
```

**Problems:**
1. **No `secure=True`** — cookie sent over HTTP, vulnerable to MITM attacks
2. **No `samesite` flag** — vulnerable to CSRF attacks
3. **Cookie max_age doesn't match server-side expiration** (cookie lasts 7 days, but server never checks)

### Recommendations

#### Fix 1: Enforce Session Expiration

```python
def get_session(db: Session, token: str) -> DBSession | None:
    now = datetime.now(tz=timezone.utc)
    stmt = (
        select(DBSession)
        .where(DBSession.token == token)
        .where(DBSession.expires_at > now)  # ✅ Check expiration
    )
    session = db.exec(stmt).first()
    
    if session:
        # Optional: Update last_active for sliding expiration
        session.last_active = now
        db.add(session)
        db.commit()
    
    return session
```

#### Fix 2: Add Session Cleanup Background Task

```python
# In cli.py or new background_tasks.py
import asyncio
from datetime import datetime, timedelta, timezone

async def cleanup_expired_sessions(engine, interval_hours=1):
    """Background task to clean up expired sessions."""
    while True:
        await asyncio.sleep(interval_hours * 3600)
        
        with Session(engine) as db:
            now = datetime.now(tz=timezone.utc)
            stmt = delete(DBSession).where(DBSession.expires_at < now)
            result = db.exec(stmt)
            db.commit()
            
            if result.rowcount > 0:
                logger.info(f"Cleaned up {result.rowcount} expired sessions")

# Start in serve_app()
@app.on_event("startup")
async def startup():
    asyncio.create_task(cleanup_expired_sessions(app.state.db_engine))
```

#### Fix 3: Secure Cookie Flags

```python
# Add configuration option
@router.post("/auth/login")
def login(...):
    # ...
    response.set_cookie(
        key=SESSION_COOKIE,
        value=token,
        httponly=True,
```

**Note:** Make `secure=True` configurable for local development (HTTP).

## Review Area 3: Concurrency & Locking Edge Cases

### Severity: **MEDIUM** ⚠️

### Issue Summary

The locking system has good fundamentals but several edge cases that could cause **race conditions**, **deadlocks**, and **resource exhaustion** under high concurrency or error conditions.

### Specific Problems

#### 3.1 Race Condition in Lock Acquisition (Lines 19-39)

**Location:** [locks.py:19-39](file:///c:/Users/cliff/projects/adapt/adapt/locks.py#L19-L39)

```python
def acquire_lock(self, resource: str, owner: str, ...) -> LockRecord:
    now = datetime.datetime.now(tz=timezone.utc)
    expires = now + datetime.timedelta(seconds=ttl_seconds)
    
    with Session(self.db_engine) as db:
        # ❌ RACE CONDITION: Check and insert are not atomic!
        existing = db.exec(select(LockRecord).where(LockRecord.resource == resource)).first()
        
        if existing and (existing.expires_at is None or existing.expires_at > now):
            raise RuntimeError(f"Resource '{resource}' is already locked by {existing.owner}")
        
        lock = LockRecord(...)  # Create new lock
        db.add(lock)
        db.commit()  # ❌ Another process could insert between check and commit!
```

**Race Condition Scenario:**
1. Process A checks for lock on `data.csv` → None found
2. Process B checks for lock on `data.csv` → None found
3. Process A inserts lock → commits
4. Process B inserts lock → commits **successfully** (no unique constraint!)
5. **Both processes think they have the lock**

**Root Cause:**  
No database-level unique constraint on the [resource](file:///c:/Users/cliff/projects/adapt/adapt/plugins/dataset_plugin.py#235-238) column, and check-then-insert is not atomic.

**Impact:**
- **Data corruption**: Two processes can write to the same file simultaneously
- Violates the entire safety guarantee of the locking system
- Will manifest under high concurrent write load

#### 3.2 Lock Not Released on Exception in Write Path

**Location:** [dataset_plugin.py:114-161](file:///c:/Users/cliff/projects/adapt/adapt/plugins/dataset_plugin.py#L114-L161)

```python
def write(self, resource: ResourceDescriptor, data: Any, request: Request, context: PluginContext):
    owner_name = owner.username if owner else "anonymous"
    
    try:
        with context.lock_manager.lock(resource.path.as_posix(), owner_name, reason=f"write:{action}"):
            existing_rows = list(self.read(resource, request))  # ❌ Could raise exception
            header = resource.metadata.get("header", [])
            
            # ... mutations ...
            
            self._write_rows(resource, existing_rows, header)  # ❌ Could raise exception
            return {"success": True}
    except RuntimeError as e:
        raise HTTPException(status_code=409, detail=str(e))
```

**Problem:**  
The code **only catches `RuntimeError`** (lock acquisition failure), but [read()](file:///c:/Users/cliff/projects/adapt/adapt/plugins/base.py#47-50) and [_write_rows()](file:///c:/Users/cliff/projects/adapt/adapt/plugins/dataset_plugin.py#245-247) can raise many other exceptions:
- `IOError` from file system
- `PermissionError` from OS
- `UnicodeDecodeError` from malformed CSV
- `ValueError` from schema validation

**Impact:**
- If non-RuntimeError exception occurs, it **propagates past the except block**
- The `with` statement still releases the lock (that's good!)
- But the **HTTP 500 error is generic, not HTTP 409 Conflict**
- Error handling is inconsistent

**Recommendation:** Broader exception handling or let all exceptions propagate as 500 (current behavior is actually correct, just confusing).

#### 3.3 No Deadlock Prevention

**Problem:**  
Multiple resources can be locked in different orders, creating potential for deadlock:

```python
# Process A
with lock_manager.lock("file1.csv", ...):
    with lock_manager.lock("file2.csv", ...):
        # work
        
# Process B
with lock_manager.lock("file2.csv", ...):  # Reversed order!
    with lock_manager.lock("file1.csv", ...):
        # DEADLOCK!
```

**Current Risk:** Low (single-file operations), **Future Risk:** High (if you add multi-file transactions)

#### 3.4 Lock Cleanup on Server Crash

**Location:** [locks.py:60-67](file:///c:/Users/cliff/projects/adapt/adapt/locks.py#L60-L67)

```python

    cutoff = datetime.datetime.now(tz=timezone.utc) - datetime.timedelta(seconds=max_age_seconds)
    with Session(self.db_engine) as db:
        stmt = delete(LockRecord).where(LockRecord.acquired_at < cutoff)
        # ...
```

**Problem:**  
The [release_stale_locks()](file:///c:/Users/cliff/projects/adapt/adapt/locks.py#60-68) method exists but is **never called automatically**. It's only exposed via the Admin UI endpoint (`/admin/locks/clean`).

**Impact:**
- If server crashes mid-write, locks remain in database
- Manual intervention required to clean up
- Files become permanently "locked" until admin cleanup
- No automatic recovery after crash

### Recommendations

#### Fix 1: Add Unique Constraint and Handle Race Condition

```python
# In storage.py
class LockRecord(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    resource: str = Field(index=True, unique=True)  # ✅ Add unique constraint
    owner: str
    # ...

# In locks.py
def acquire_lock(self, resource: str, owner: str, ...) -> LockRecord:
    now = datetime.datetime.now(tz=timezone.utc)
    expires = now + datetime.timedelta(seconds=ttl_seconds)
    
    with Session(self.db_engine) as db:
        try:
            # Try to insert directly (optimistic)
            lock = LockRecord(resource=resource, owner=owner, acquired_at=now, expires_at=expires, reason=reason)
            db.add(lock)
            db.commit()
            db.refresh(lock)
            return lock
        except IntegrityError:  # ✅ Unique constraint violation
            db.rollback()
            # Lock exists, check if expired
            existing = db.exec(select(LockRecord).where(LockRecord.resource == resource)).first()
            if existing and (existing.expires_at is None or existing.expires_at > now):
                raise RuntimeError(f"Resource '{resource}' is already locked by {existing.owner}")
            else:
                # Expired lock, delete and retry
                db.delete(existing)
                db.commit()
                return self.acquire_lock(resource, owner, reason, ttl_seconds)
```

#### Fix 2: Add Automatic Lock Cleanup on Startup

```python
# In cli.py serve_app()
def serve_app(config: AdaptConfig):
    # ... initialization ...
    
    # Clean up stale locks from previous crash
    lock_manager = LockManager(engine)
    cleaned = lock_manager.release_stale_locks(max_age_seconds=300)  # 5 minutes
    if cleaned > 0:
        logger.warning(f"Cleaned {cleaned} stale locks on startup")
    
    app.state.lock_manager = lock_manager
```

#### Fix 3: Add Lock Timeout to Context Manager

```python
class _LockContext:
    def __init__(self, manager, resource, owner, reason=None, timeout_seconds=30):
        self.timeout = timeout_seconds
        # ...
    
    def __enter__(self):
        start = time.time()
        while time.time() - start < self.timeout:
            try:
                self.lock = self.manager.acquire_lock(...)
                return self.lock
            except RuntimeError:
                time.sleep(0.1)  # Backoff
        raise TimeoutError(f"Failed to acquire lock on {self.resource} after {self.timeout}s")
```

### Testing Recommendations

1. **Race condition test**: Spawn 10 threads trying to acquire same lock simultaneously
2. **Crash recovery test**: Kill server mid-write, restart, verify lock is cleaned
3. **Lock expiration test**: Acquire lock with TTL=1s, wait 2s, verify auto-release
4. **Unique constraint test**: Verify database enforces lock uniqueness

## Conclusion

The Adapt codebase demonstrates solid engineering practices and architectural thinking. The authentication system is well-designed, the plugin architecture is clean, and the locking mechanism shows good intent. However, **the devil is in the details**—session lifecycle bugs, race conditions, and resource management issues could cause significant problems in production.

**Key Takeaway:**  
Focus on **session management** and **database transaction handling** before deploying to production. These are foundational issues that affect stability, security, and data integrity.

The good news is that all identified issues have clear solutions and the architecture supports the necessary fixes without major refactoring.
