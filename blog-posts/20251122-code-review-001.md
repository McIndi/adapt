# Code Review: Adapt

## Overview
This review analyzes the current state of the Adapt codebase (v0.1.0) with a specific focus on readiness for the upcoming "Users, Groups, and Permissions" features. The review covers architecture, security, performance, and maintainability.

## Findings

### 1. Architecture & Design

#### **Plugin-Controlled Routing vs. Global Security**
*   **Observation**: The recent refactor moves route generation to plugins via `get_route_configs`. Plugins return fully configured `APIRouter` instances.
*   **Issue**: This decentralization makes it difficult to enforce a global security policy. Since plugins own the routers, the core server cannot easily inject authentication dependencies or wrap endpoints in permission checks without modifying the `Plugin` interface or using global middleware.
*   **Risk**: Retrofitting granular permissions (e.g., "User A can only read Sheet 1") will be challenging if the routing logic is opaque to the core server.

#### **State Management**
*   **Observation**: Plugins are instantiated on-the-fly (`plugin = plugin_cls()`) in `discovery.py` and `routes.py`.
*   **Issue**: Plugins are effectively stateless. While this is generally good, it prevents optimization strategies like keeping file handles open or caching expensive setup steps within the plugin instance.
*   **Risk**: Performance overhead during startup and potentially during request handling if plugins do heavy initialization.

### 2. Security & Permissions

#### **Missing Authentication Layer**
*   **Observation**: `storage.py` defines `User` and `Permission` models, and `cli.py` allows creating a superuser. However, there is **no authentication middleware** or dependency wired into the FastAPI app.
*   **Issue**: The server is currently completely open. The existing "superuser" is effectively a ghost record in the database with no power.
*   **Risk**: Implementing auth requires a significant architectural addition (e.g., OAuth2 with JWT) that needs to be threaded through *all* plugin routes.

#### **Coarse-Grained Permissions**
*   **Observation**: The `Permission` model consists of `resource` and `action`.
*   **Issue**: This model is likely too simple for complex datasets. It doesn't account for:
    *   **Row-level security**: "User X can only see rows where Department=Sales".
    *   **Column-level security**: "User Y cannot see the 'Salary' column".
    *   **Hierarchical resources**: Permissions for `/api/workbook` vs `/api/workbook/Sheet1`.
*   **Risk**: The current model will force a binary "all or nothing" access control scheme.

#### **Missing Write Locking**
*   **Observation**: `storage.py` defines `LockRecord`, but `DatasetPlugin.write` does not appear to acquire or check locks.
*   **Issue**: Concurrent writes could corrupt data files. The spec promises "Safe writes via file locking", but the implementation is missing.
*   **Risk**: Data corruption in multi-user environments.

### 3. Scalability & Performance

#### **O(N) Write Operations**
*   **Observation**: `DatasetPlugin.write` reads *all* rows into memory, applies changes, and writes *all* rows back to disk.
*   **Issue**: This is O(N) with respect to file size. For large datasets (e.g., 100k+ rows), this will be extremely slow and memory-intensive.
*   **Risk**: The server will become unresponsive or crash with OOM errors on moderately large files.

#### **Re-Reading Files**
*   **Observation**: `DatasetPlugin.read` re-reads the file from disk on every request.
*   **Issue**: No caching of file contents.
*   **Risk**: High I/O latency, especially for Excel files which are slow to parse.

### 4. Maintainability

#### **Coupling to FastAPI**
*   **Observation**: `Plugin` methods (`read`, `write`) accept `fastapi.Request` objects.
*   **Issue**: This tightly couples the plugin logic to the HTTP layer. It makes it harder to test plugins in isolation (without a mock Request) or to use plugins in a non-HTTP context (e.g., a CLI tool or background worker).
*   **Recommendation**: Decouple core logic from the web framework. Pass strict data structures instead of `Request`.

## Recommendations

### 1. Implement Global Auth Middleware
Instead of relying on plugins to secure themselves, implement a **Global Authentication Middleware** in `adapt/security.py` (to be created).
*   This middleware should intercept all requests to `/api/*`, `/ui/*`, and `/schema/*`.
*   It should resolve the current user from a session token or API key.
*   It should attach the user to `request.state.user`.

### 2. Standardize Permission Enforcement
Create a `SecurityContext` that is passed to plugins, or enforce a standard dependency on all plugin routes.
*   **Option A (Middleware)**: The middleware checks permissions based on the URL path (mapping path -> resource). This is fragile if URLs change.
*   **Option B (Dependency)**: Update `Plugin.get_route_configs` to accept a `security_dependency`. Plugins *must* include this dependency in their routers.

### 3. Refactor Write Logic for Locking
*   Update `DatasetPlugin.write` to:
    1.  Acquire a lock via `storage.LockManager` (needs implementation).
    2.  Perform the read-modify-write cycle.
    3.  Release the lock.
*   Ensure this is atomic.

### 4. Enhance Permission Model
*   Update `Permission` model to support wildcards (e.g., `resource="workbook/*"`) or hierarchy.
*   Consider adding `filter` or `mask` fields for row/column security.

### 5. Decouple Plugins from Request
*   Change `read(resource, request)` to `read(resource, query_params, user)`.
*   This makes testing easier and clarifies the data dependencies.

## Conclusion
The current codebase is a solid prototype but requires significant architectural work to support secure, multi-user usage. The priority should be **implementing the missing security layer** and **enforcing locking** before adding more features.
